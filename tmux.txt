Ninereeds Codex tmux/status setup
==================================

Purpose
-------

Run Codex in a named tmux session so the MSM watchdog can passively read visible Codex
status output and write:

- training/pipeline/msm/state/codex_pane_snapshot.txt
- training/pipeline/msm/state/codex_status.json
- training/pipeline/msm/state/codex_status.md
- training/pipeline/msm/state/codex_brake.json

The watchdog must not inject commands into Codex. It only reads the tmux pane.


1. Confirm tmux exists
----------------------

From a normal shell:

```bash
tmux -V
```

If this fails, install tmux first.


2. Start Codex in tmux
----------------------

From a normal shell:

```bash
cd ~/Ninereeds
tmux new -s codex 'codex'
```

If a `codex` tmux session already exists:

```bash
tmux attach -t codex
```

Detach from tmux without closing Codex:

```text
Ctrl-b then d
```

Do not close the tmux window unless you want to stop the Codex session.


3. Configure Codex status display
---------------------------------

Inside Codex, use `/statusline` manually to configure what `/status` shows.

The visible `/status` display should include at least:

- model / reasoning setting
- directory
- session id
- context window
- 5h limit
- weekly limit
- reset times

Known-good status lines look like:

```text
Context window:       75% left (74.1K used / 258K)
5h limit:             [██████████████████░░] 90% left (resets 06:03)
Weekly limit:         [██████████████████░░] 89% left (resets 15:24 on 8 Jul)
```

Important:

- `/status` and `/statusline` are Codex TUI commands, not shell commands.
- The watchdog must not send `/status` or `/statusline` into the live Codex session.
- Configure this manually, then leave the status output visible often enough for tmux
  capture to include the rate-limit lines.


4. Test passive capture
-----------------------

From another shell, while the `codex` tmux session is running:

```bash
tmux capture-pane -t codex -p -S -120 | tail -40
```

Confirm the output contains lines like:

```text
5h limit:             ... 90% left (resets 06:03)
Weekly limit:         ... 89% left (resets 15:24 on 8 Jul)
```

If those lines are not present, return to Codex, show `/status` manually, then detach
again and retry the capture command.


5. Run the watchdog once
------------------------

From a normal shell:

```bash
cd ~/Ninereeds
python3 meta/scripts/watch_codex_status.py
```

Expected success exit: 0

Read the outputs:

```bash
cat training/pipeline/msm/state/codex_status.md
cat training/pipeline/msm/state/codex_brake.json
```

Expected brake action while usage is low:

```json
"action": "continue"
```


6. If the watchdog cannot parse status
--------------------------------------

If the command exits 2 or writes `blocked_unknown_reset`, inspect:

```bash
cat training/pipeline/msm/state/codex_pane_snapshot.txt
cat training/pipeline/msm/state/codex_status.json
cat training/pipeline/msm/state/codex_brake.json
find training/pipeline/msm -name BLOCKED -print
```

Common cause:

- The visible tmux pane did not contain the `5h limit` or `Weekly limit` lines.

Fix:

1. Attach to Codex:

```bash
tmux attach -t codex
```

2. Show `/status` manually.
3. Detach with `Ctrl-b then d`.
4. Rerun:

```bash
cd ~/Ninereeds
python3 meta/scripts/watch_codex_status.py
```

If the snapshot contains the limit lines but parsing still fails, do not edit pipeline
state by hand. Ask Codex to adjust `meta/scripts/watch_codex_status.py` to match the
actual visible status text.


7. Optional manual loop before cron
-----------------------------------

Before enabling cron, you can run a simple temporary loop in a normal shell:

```bash
cd ~/Ninereeds
while true; do
  date
  python3 meta/scripts/watch_codex_status.py || true
  cat training/pipeline/msm/state/codex_brake.json
  sleep 300
done
```

Stop it with `Ctrl-c`.


8. Later cron handoff
---------------------

Once the one-shot watchdog test works, Hermes can install or verify cron/systemd entries.
The intended deterministic watchdog command is:

```bash
cd ~/Ninereeds && python3 meta/scripts/watch_codex_status.py
```

Recommended cadence:

```text
Every 1-5 minutes: refresh Codex status/brake artifacts.
Every 1 hour: Hermes reads artifacts and sends a compact status digest.
```

Hermes should read and report. It should not edit MSM plans, policy, queues, update
manifests, corpus files, checkpoints, or concept state.
