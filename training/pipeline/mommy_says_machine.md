# Mommy Says Machine

Mommy Says Machine is the active training loop for Ninereeds.
For executable step order, use `runbook.md`; this file defines the system boundary.

Older docs described MSM as a diagnostic intervention around traditional corpus training.
That role is deprecated. MSM now means session-based teaching, reporting, repair, and
optional micro-updates.

---

## Purpose

MSM answers:

1. What does Ninereeds currently do when asked about this word?
2. Which axes are stable, weak, or wrong?
3. Which correction shape helps?
4. Which proposed turns are safe enough for orchestrator review?
5. Should the next step be replay, repair, probe, brain scan, update, or escalation?

---

## Machine Boundary

MSM is a pipeline, not one model.

- Gemma executes fixed scripts.
- DeepSeek writes scripts and reports.
- The orchestrator decides strategy.
- Hermes reports status and pings the user when needed.

This separation is intentional. The worker that runs prompts does not grade them, and
the worker that grades logs does not own the research plan.

---

## Session Structure

Each session targets one word/card.

Inputs:

- parent checkpoint
- concept card
- orchestrator plan
- script generated by DeepSeek

Outputs:

- raw chat log
- fixed report card
- turn-level grades
- failed-turn diagnosis
- optional proposed training turns

---

## Mommy Says Correction Rule

Corrections:

- restate the correct form
- avoid praise and judgment
- are short
- target one failure mode at a time
- use known support vocabulary when possible
- are retested immediately

Example:

```text
[user] A cat is an animal. A dog is an animal. A cat is not a dog.
```

Do not train from the wrong Ninereeds answer.

---

## Report-First Design

DeepSeek must fill `report_card.json` after every session. The orchestrator should be able
to compare sessions without parsing inconsistent prose.

Required schema: `training/pipeline/session_report_schema.md`.

---

## Closed Loop

MSM can update weights, but only through orchestrator-approved turns:

1. Raw session is logged.
2. DeepSeek grades turns.
3. DeepSeek proposes grading-validated corrections/expected answers.
4. The orchestrator approves selected turns into an update buffer.
5. A buffered micro-update creates an update-candidate checkpoint from approved turns.
6. Target checks and protected anchors decide acceptance.

Raw logs are never ingested directly. DeepSeek-proposed turns are not ingested until the
orchestrator copies approved records into `approved_training.jsonl`.

---

## Rollback-Replay

If a session reveals harmful confusion, the default repair is rollback-replay:

1. Return to the pre-session checkpoint.
2. Generate a repair replay script for the failure mode.
3. Execute and report the replay.
4. Accept only if the repair holds and protected anchors pass.

This is the standard response to failures such as:

- `a cat is a dog`
- `a tree is a mammal`
- identity/name confusion
- unknown-boundary failure
- topic drift that becomes stable

---

## Cold Start

Bootstrapped MSM begins from:

`core/c17_contrast_angle_1200_e4.pt`

Cold-start MSM from random weights is allowed only as a distinct experiment. Its logs may
begin with byte noise, letter fragments, malformed words, and nonsensical sentences.
Cold-start reporting should still use the same schema, but thresholds must be different.

---

## Human Attention

The orchestrator or Hermes writes a sentinel file when automation cannot continue:

- machine crashed
- script execution failed repeatedly
- API key/credits failed
- report card invalid
- promotion decision needs user judgment
- unexpected damage to protected anchors

Hermes watches for sentinel files and pings the user on Discord.
